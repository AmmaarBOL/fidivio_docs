# Program Setup

Navigate to **Setup & Support → Configuration → Program Setup** to configure the key attributes and financial parameters of your loyalty program. This includes program details, currency settings, opening balances, and accounting rules.

---

## 3.1 Program Info

This section defines the core details of your loyalty program.

| Field | Description | Format | Required |
|---|---|---|---|
| **Program Name** | Name of your loyalty program. | Text | Yes |
| **Program Type** | Select the program model: Profit Centre Model or Conventional Model. | Selection | Yes |
| **Start Date** | The date the program commenced. | Date | Yes |
| **End Date** | The date the program ends, if applicable. | Date | Optional |
| **Country** | Country where the program operates. | Selection | Yes |
| **Timezone** | System timezone used for reporting and calculations. | Selection | Yes |
| **Program Currency Name** | Name of your loyalty currency (e.g. Points, Miles, Credits). | Text | Yes |
| **Monetary Currency Code** | Currency used for financial reporting (e.g. USD, EUR, MYR). | ISO 4217 | Yes |
| **Fiscal Year Start Month** | Month in which your financial year begins. | Selection | Yes |
| **Program / Company Logo** | Logo of the program or company, displayed across the platform. | Image upload | Optional |

---

## 3.2 Currency Settings

This section controls how monetary values and dates are displayed and calculated across the platform.

| Field | Description | Format | Required |
|---|---|---|---|
| **Currency Format** | Defines number formatting for monetary values (e.g. $1,234.56). | Selection | Yes |
| **Date Format** | Format used to display dates across the platform (e.g. DD/MM/YYYY). | Selection | Yes |
| **Decimal Precision for Calculation** | Number of decimal places used in backend model calculations. | Numeric | Yes |
| **Decimal Precision for Display** | Number of decimal places shown in dashboard metrics and reports. | Numeric | Yes |

## 3.3 Existing Program Balance

Use this section when migrating data from an existing loyalty program to Fidivio to set opening balances.

<Warning>
Leave these fields blank if you are setting up a new program with no prior history. Entering incorrect opening balances will affect all subsequent deferred revenue and liability calculations.
</Warning>

| Field | Description | Format | Required |
|---|---|---|---|
| **Opening Points Balance** | Total outstanding points at the beginning of the first data year — i.e. the cumulative points issued but not yet redeemed prior to the first data year. | Numeric | Optional |
| **Opening Redeemed Points** | Number of points from the opening balance that are expected to ultimately be redeemed (must be ≤ total opening balance). Used to determine the breakage estimate applied to the opening balance. | Numeric | Optional |
| **Opening Deferred Revenue Balance** | The deferred revenue (contract liability) balance carried over from the prior program, corresponding to the monetary value assigned to the opening points balance. | Numeric | Optional |

## 3.4 Accounting Configuration

This section defines the preferred accounting treatments for the program. These choices directly affect how revenue is reported on the income statement. There are three configurable settings:

- **Setting 1:** Applied Standalone Selling Price (SSP) — Is the SSP adjusted to reflect expected breakage?
- **Setting 2:** Breakage Recognition — How and when is breakage recognized as revenue?
- **Setting 3:** Deferred Revenue True-up — How is deferred revenue adjusted for changes in estimates?

Together these three settings control the revenue recognition and adjustment logic of the model.

<Warning>
Accounting settings should be established at program set-up and applied consistently thereafter. Changing settings mid-program will result in the latest configuration being applied retrospectively to all historical data.
</Warning>

---

### Setting 1 — Applied Standalone Selling Price (SSP)

The Standalone Selling Price (SSP) of a loyalty point represents what the program operator would charge if selling that point separately to a customer. In Fidivio, the SSP data input is defined as the gross face value of one point, before any adjustment for expected breakage. A key accounting decision is whether the applied SSP used in the model should remain at this gross figure or be reduced to reflect the proportion of points not expected to be redeemed. This choice — whether breakage is embedded in the applied SSP or handled separately — directly affects how much revenue is deferred as a contract liability versus recognised immediately on the income statement.

| Option | Description | Example |
|---|---|---|
| **Yes (Default)** | The applied SSP is the SSP input after breakage has been deducted. The model uses this reduced figure to value performance obligations. In most real-world programs the SSP is estimated with breakage embedded, reflecting the expected redemption value weighted by the probability that a point will actually be redeemed. This is the recommended default. | SSP: $0.01; Breakage: 20% → Applied SSP = $0.01 × 80% = $0.008 |
| **No** | The applied SSP is taken directly from the SSP input without any adjustment for breakage. The model uses the gross face value of a point. This is a more conservative approach to liability provisioning — the full face value of every point is held as a liability regardless of expected breakage. | SSP: $0.01; Breakage: 20% → Applied SSP = $0.01 |

---

### Setting 2 — Breakage Recognition

Breakage refers to the portion of loyalty points that members are not expected to redeem. The Breakage Recognition setting determines the timing and method of revenue recognition for these unexercised rights. In general, breakage income should be released progressively over time so the liability balance better reflects the remaining obligation.

| Setting | Description | When to use |
|---|---|---|
| **1 (Default) — In proportion to redemption** | Breakage revenue is recognized in proportion to redemption. As points are redeemed each period, a proportionate share of expected breakage is also recognized. Breakage income = (Points redeemed in period ÷ Total points expected to be redeemed) × Total breakage revenue. | The IFRS 15 default method. Recommended for all programs, especially those where points do not expire or where expiry is triggered by account inactivity rather than a fixed date. |
| **2 — In proportion to validity period** | Breakage revenue is recognized in proportion to the validity period. The model spreads expected breakage evenly over the points' validity window (e.g. 12 months) regardless of redemption activity. The validity period (in months) must be configured in the data input. | Use when the program has explicit point expiry dates and expiry is the primary driver of breakage. A breakage reconciliation input is required to account for any difference between estimated and actual expiry outcomes. |
| **3 — On actual expiry events** | Breakage revenue is recognized based on actual expiry events. Revenue is only recognized when points are actually expired and removed from member accounts. | Use when the entity cannot reliably estimate breakage in advance, or when the operator prefers a conservative policy. Avoids uncertainty in breakage estimation but may result in more volatile revenue recognition. |

---

### Setting 3 — Deferred Revenue True-up

As time passes, a loyalty program operator develops new information about how members will ultimately redeem their points. Two estimates can change over time: the breakage rate, and the standalone selling price (SSP). When either changes, the contract liability for all outstanding points may need to be restated. The Deferred Revenue True-up setting determines whether and how such catch-up adjustments are applied.

| Setting | Description | Principle / Implication |
|---|---|---|
| **1 (Default) — No true-up** | No true-up for any change in breakage outlook or SSP. The contract liability is fixed at the amount of revenue deferred when consideration is first received. A manual adjustment can be made for discretionary inputs under Other Revenue Sources. | When used alongside Setting 1 (breakage in proportion to redemption), this model naturally adjusts to changes in breakage rates over time, minimizing one-off income statement impacts. Revenue recognized per point = total deferred revenue ÷ total outstanding points. |
| **2 — True-up for breakage changes** | When the breakage rate is revised (using breakage adjustment input), the model automatically calculates a cumulative catch-up adjustment to restate the outstanding point liabilities to reflect the updated estimate. The difference flows to the income statement in the current period. | Select when you want the model to automatically restate the liability whenever a breakage adjustment happens. A higher breakage estimate = release of liability (positive P&L); a lower estimate = increase in liability (negative P&L). This setting is not applicable when the applied SSP is not adjusted for breakage. |
| **3 — True-up for SSP changes** | When the applied SSP changes, the model automatically restates the liability for all outstanding points to reflect the updated SSP. Every point still in circulation is remeasured at the new SSP. The difference flows to the income statement as a catch-up adjustment. | Use when you want the liability to always reflect the current best estimate of the SSP for all outstanding points, regardless of when they were issued. |

---

### Income Statement Impact Summary

The three accounting settings work together to determine the full IFRS income statement.

| Income Statement Line | Driven By |
|---|---|
| Revenue recognized (redemptions) | Affected by all 3 settings. The Applied SSP setting, Breakage Recognition setting, and Deferred Revenue True-up setting all influence the effective revenue rate per point recognised. |
| Breakage income | Timing is controlled by the Breakage Recognition setting. Revenue recognised per point is also influenced by the Applied SSP setting and the Deferred Revenue True-up setting. |
| Marketing income | Driven by the Applied SSP setting. Arises when the yield per point exceeds the applied SSP. When SSP is set to post-breakage (Yes), marketing income tends to be higher because the applied SSP baseline is lower. Does not apply to programs run under the conventional model. |
| Adjustment for liability true-up | Driven exclusively by the Deferred Revenue True-up setting. This line is zero when Setting 1 (no true-up) is selected. |
