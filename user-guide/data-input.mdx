The Data Input section (accessible via **Setup & Support → Data Input**) is where actual and budget data are uploaded into the model. Data can be entered manually or imported in bulk via CSV batch upload.

---

## 6.1 How to Add Data

1. Under **Setup & Support**, click on **Data Input** and select the data category.
2. Navigate to the relevant data tab (**Actual** or **Budget**).
3. Click the **+ Add** icon to open the entry panel.
4. Choose an input method:
   - **Manual** — complete the on-screen form fields and click **Add**
   - **Upload** — download the CSV template for that category, populate it, and upload the file

---

## 6.2 Batch File Upload (CSV)

Each data category has its own downloadable CSV template. All CSV files share a common structure:

- The file must be saved in CSV format (`.csv`).
- The first column is always `action` — specifies whether the row is an insert, update, or delete.
- The second column is always `id` — the unique record identifier managed by Fidivio.
- All remaining columns are category-specific data fields.

| Action Value | Description |
|---|---|
| `insert` | Adds a new record. The `id` column should be left blank. |
| `update` | Overwrites an existing record. The `id` of the record to update must be provided. |
| `delete` | Removes a record. The `id` of the record to delete must be provided. |

<Note>
The `id` value for each record is auto-generated by Fidivio at the time of insertion. To obtain the `id` for an existing record (for update or delete), export the current dataset from the relevant tab.
</Note>

---

## 6.3 Common Data Fields

The following four fields appear in every data category template:

| Field | Definition | Format | Required |
|---|---|---|---|
| `action` | Specifies the operation: insert, update, or delete. | `insert` \| `update` \| `delete` | Yes |
| `id` | Unique record identifier auto-generated by Fidivio. Required for update and delete. | Integer | Conditional |
| `transaction_date` | The date associated with the data record. | `yyyy-mm-dd` | Yes |
| `external_unique_identifier` | Optional reference ID defined by the user for cross-system tracking. | Integer | Optional |

---

## 6.4 Data Categories

There are seven data categories available for input:

| # | Category | What it captures |
|---|---|---|
| 1 | **Accruals** | Points issued to members and the corresponding yield per point earned by the program. |
| 2 | **Breakage Levels** | Breakage rate assumptions, SSP inputs, validity periods, and adjustment figures. |
| 3 | **Expiry** | Actual points removed from member accounts due to expiry events in a given period. |
| 4 | **Other Revenue Sources** | Non-points revenue earned by the program (e.g., brand licensing, management fees). |
| 5 | **Program Costs** | Operational and administrative costs of running the loyalty program. |
| 6 | **Redemptions** | Points redeemed by members and the associated cost per point incurred. |
| 7 | **Sales Information** | Contract liability data for goods sold that earned points (only applicable if Program Type is set as Conventional model). |

---

### 6.4.1 Accruals

Accruals record the number of loyalty points issued to members in each period, alongside the yield (revenue) earned by the program for issuing those points. Each row represents a single transaction date and partner combination.

**When to use:** Enter an accrual record for every period in which points are awarded to members. Where multiple partners issue points on the same date, enter a separate row for each partner. Yield is required for profit-centre programs; enter `0` for self-funded programs.

<Note>
Partner codes must be set up under **Setup & Support → Partners** before accrual data can be uploaded.
</Note>

| Field | Definition | Format | Required |
|---|---|---|---|
| `partner_code` | Code identifying the partner responsible for this accrual. Must match a configured partner code exactly. | Text | Yes |
| `currency` | The currency in which the yield is denominated. This should match the selected currency in Program Setup. | ISO 4217 (e.g. USD, EUR) | Yes |
| `points_issued` | Total number of loyalty points issued on this transaction date for this partner. | Integer, no decimal | Yes |
| `yield` | Revenue earned per point issued (price per point charged to partner). Set to `0` for self-funded programs. | Numeric, decimal allowed | Yes |

---

### 6.4.2 Breakage Levels

Breakage Levels hold the key actuarial and accounting assumptions used by the model to estimate how many points will ultimately not be redeemed. These inputs drive the applied SSP, the contract liability balance, and the breakage income recognised each period.

**When to use:** Enter one row per transaction date to set the breakage assumptions for that period's point issuances. Leave `breakage_adjustment` and `breakage_reconciliation` as `0` if not applicable.

| Field | Definition | Format | Required |
|---|---|---|---|
| `breakage_level` | Expected % of points issued in this period that will not be redeemed. E.g. `20` = 20%. | Number 0–100 | Yes |
| `breakage_adjustment` | Manual adjustment to breakage estimate for all outstanding points. Positive = more expected to break; negative = fewer. In points. | Numeric, +/− | Optional |
| `breakage_reconciliation` | Difference between breakage estimate and observed actual expiry. Used to reconcile model vs reality when breakage is recognized in proportion to validity period (refer to Accounting Configuration → Breakage Recognition setting). In points. | Numeric, +/− | Optional |
| `points_validity` | Number of months for which points issued in this period remain valid. Required when Breakage Recognition is set to recognise by validity period. | Integer (months) | Conditional |
| `stand_alone_selling_price` | SSP of one point for this period (pre-breakage face value). The applied SSP used in the model depends on your Applied SSP configuration setting. | Numeric (e.g. 0.0080) | Yes |

---

### 6.4.3 Expiry

Expiry records capture the actual number of points voided and removed from member accounts in each period. These are used when Breakage Recognition is set to Setting 3 (actual expiry events) and for points movement tracking in the Points Overview Dashboard.

**When to use:** Enter an expiry record in any period where the program formally expires points. No entry is needed if no points expired in a given period.

| Field | Definition | Format | Required |
|---|---|---|---|
| `points_expired` | Total number of points formally expired and removed from member accounts on this transaction date. | Numeric ≥ 0 | Yes |

---

### 6.4.4 Other Revenue Sources

Other Revenue Sources records non-points income earned by the loyalty program — for example, brand licensing, co-marketing arrangements, access to member data, or program management fees. Unlike points-related revenue, this income is typically recognised at the time it is received.

**Pre-configuration required:** `revenue_category` and `revenue_item` must be set up before uploading data.

To create a new category or item:
1. Go to **Add → Manual**.
2. Under **Category**, type the category name into the Search box and click **Create New**.
3. Repeat under **Item** to create the corresponding item entry for the selected category.

<Warning>
These values must be created first — they cannot be entered during batch data upload.
</Warning>

| Field | Definition | Format | Required |
|---|---|---|---|
| `non_currency_revenue` | Monetary value of non-points revenue received in this period. | Numeric ≥ 0 | Optional |
| `revenue_category` | High-level revenue category (e.g. Brand & Marketing). Must match a pre-configured category. | Text | Optional |
| `revenue_item` | Specific revenue item within the category (e.g. Annual Brand Fee). Must match a pre-configured item. | Text | Optional |
| `currency` | Currency in which the revenue amount is denominated. This should match the selected currency in Program Setup. | ISO 4217 code | Optional |

---

### 6.4.5 Program Costs

Program Costs records the running costs of operating the loyalty program, excluding the direct cost of fulfilling redemptions (which is captured under Redemptions). Typical examples include staff costs, technology licences, office expenses, and marketing spend.

<Note>
Program Costs captures only operational overheads. The direct cost of fulfilling member redemptions is entered separately under the Redemptions category.
</Note>

**Pre-configuration required:** `cost_category` and `cost_item` must be set up before uploading data.

To create a new category or item:
1. Go to **Add → Manual**.
2. Under **Category**, type the category name into the Search box and click **Create New**.
3. Repeat under **Item** to create the corresponding item entry for the selected category.

<Warning>
These values must be created first — they cannot be entered during batch data upload.
</Warning>

| Field | Definition | Format | Required |
|---|---|---|---|
| `cost_category` | High-level cost category (e.g. Staff Costs, Technology). Must match a pre-configured category. | Text | Optional |
| `cost_item` | Specific cost item within the category (e.g. Platform Licence Fee). Must match a pre-configured item. | Text | Optional |
| `currency` | Currency in which the cost is denominated. This should match the selected currency in Program Setup. | ISO 4217 code | Optional |
| `total_running_costs` | Total cost incurred for this cost item in this period. Must be ≥ 0. | Numeric | Optional |

---

### 6.4.6 Redemptions

Redemptions records the points that members have redeemed in a given period, along with the actual cost incurred to fulfil those redemptions. Together these determine the total redemption cost recognised in the income statement.

**When to use:** Enter a record for every period in which members redeem points. Where different redemption types or partners have materially different costs per point, enter a separate row for each combination.

<Tip>
Cost per point should reflect the actual blended cost for the period and redemption type — not the retail value of the reward. Where cost varies significantly across redemption types, use separate rows to maintain accurate cost tracking.
</Tip>

| Field | Definition | Format | Required |
|---|---|---|---|
| `redemption_type` | Label describing the type of redemption (e.g. Flight Award, Retail Voucher, Cash Redemption). Used for reporting. | Text | Yes |
| `redemption_partner_code` | Code of the partner through whom the redemption was fulfilled. Must match a configured partner code. | Text | Yes |
| `currency` | Currency in which the redemption cost is denominated. This should match the selected currency in Program Setup. | ISO 4217 code | Yes |
| `cost_per_point` | Average cost incurred by the program to fulfil the redemption of one point for the period. = Total cost ÷ Points redeemed. | Numeric ≥ 0 | Yes |
| `points_redeemed` | Total number of points redeemed by members in this period for this redemption type and partner. | Integer ≥ 0 | Yes |

---

### 6.4.7 Sales Information

Sales Information is relevant only for programs where the operator is not run as a profit centre (**Program Type = Conventional Model**). The inputs are used to determine the standalone selling price of goods associated with a transaction and the total consideration received, enabling the transaction price allocation between goods and points performance obligations.

**When to use:** Only required when Program Type is set to Conventional Model. Leave empty for profit-centre programs. Enter one row per transaction date representing the aggregated position for that period.

<Info>
The allocation of consideration between goods and points is calculated automatically by the model using the relative standalone selling price method (IFRS 15). The SSP of the points is sourced from the Breakage Levels data for the same transaction date.
</Info>

| Field | Definition | Format | Required |
|---|---|---|---|
| `partner_code` | Code of the partner for whom this sales transaction applies. Must match a configured partner code. | Text | Conditional |
| `contract_liability_associated_goods` | Standalone selling price of the goods or services sold in the same period the partner's points were issued. Represents the SSP of the initial performance obligation before consideration is allocated to points. | Numeric ≥ 0 | Conditional |
| `total_consideration_goods_and_points` | Total amount of consideration received from the customer for both goods and points entitlement combined. This is the total transaction price to be allocated between the two performance obligations. | Numeric ≥ 0 | Conditional |

## 6.5 Checking Upload Status

After submitting a batch file upload, you can monitor its processing status without leaving the platform.

1. Navigate to the relevant data category tab (e.g. Accruals, Redemptions).
2. Click the **Batches** tab at the top of the data category screen. This tab lists all batch uploads submitted for that category, along with their submission date, record count, and current status.
3. Locate your upload in the list. Statuses include:
   - **Processing** — upload is being validated and ingested
   - **Success** — all records imported successfully
   - **Failed with Errors** — one or more records could not be imported
4. If the status shows **Failed with Errors**, click the **view (eye) icon** on that batch row. A detail panel will open showing the total number of records, how many succeeded, and how many failed. From this panel you can download the original input file and the handback file.

<Tip>
Always check the Batches tab after a large upload before proceeding to view reports.
</Tip>

---

## 6.6 Data Upload Error

When errors are detected in the data file, the upload will fail and no data will be processed until all errors are resolved.

To identify and fix the error(s):

1. Go to the specific data category under **Data Input**.
2. Go to the **Batches** tab, select the file name in the upload list, click the **View** icon.
3. **Download handback file.** The handback file is a copy of your original CSV with an additional column highlighting the specific error for each failed row.
4. Review the file, correct the errors in the affected rows, and re-upload the updated file as a new batch.

---

## 6.7 Exporting Data

All data can be exported as a CSV from each data category page. Click **Export** in the top right corner, select the desired data range in the on-screen form, and click **Export** again.
